# 2 总体技术方案

## 2.2 技术架构与技术路线

### 2.2.1 技术架构选型

本项目以微服务架构、流批一体化框架和B/S架构为核心，构建覆盖模型训练、平台服务和场景输出的完整技术底座。三种架构模式相互配合，支撑沙尘预测AI模型从训练到推理的全链路能力。

**微服务架构**按业务边界将系统拆分为独立服务单元，各服务独立部署、独立扩展，通过服务注册与发现机制实现协调。本项目将预测服务、实况监测服务、预警服务、场景服务、GIS服务、接口服务等拆分为独立微服务，各服务使用容器化部署，支撑动态扩缩容，响应REQ-S4-01（微服务架构）。

微服务拆分原则如下表所示：

| 服务名称 | 职责边界 | 主要依赖 | 扩展方式 |
|---------|---------|---------|---------|
| 预测服务 | 模型推理，接收融合特征，输出风险等级 | 融合特征存储、模型存储 | 根据推理调用量水平扩展副本数 |
| 实况监测服务 | 采集气象站/监测站/遥感数据，写入实时库 | 外部数据源API | 根据监测点数量垂直扩展 |
| 预警服务 | 阈值判断、预警制作、审核、发布记录 | 预测服务、配置库 | 根据预警生成频率扩展 |
| 场景服务 | 生成交通/健康/旅游场景建议 | 预警服务、GIS服务 | 根据调用渠道数量扩展 |
| GIS服务 | 空间查询、圈选、等值线分析 | 空间数据库 | 根据并发请求量扩展 |
| 用户管理服务 | 用户注册、认证、权限管理 | 关系数据库 | 垂直扩展（无状态） |
| API网关服务 | 统一鉴权、限流、日志、路由 | 各内部服务 | 水平扩展，前置负载均衡 |

**流批一体化框架**将实时数据处理与离线批处理统一管理。气象站、监测站、能见度等实时数据通过流式管道处理，模型推理的批量预测和历史回溯通过批处理管道执行，两套管道共享特征存储和模型存储层，统一由流批一体处理引擎调度，响应REQ-S4-02（流批一体化框架）。

流批一体实现要点：
- 统一任务调度器（Flink）管理流式作业和批式作业的依赖关系；
- 两套管道共用同一套特征工程代码，保证实时和离线特征口径一致；
- 批处理任务复用流处理管道的质量控制逻辑，避免重复清洗；
- 故障恢复机制：流处理管道配置checkpoint，批处理任务完成后写结果到分布式存储。

**B/S架构**采用浏览器/服务器模式，用户通过Web管理端和微信小程序访问平台服务。服务端提供JSON over HTTPS标准API，支撑跨平台、跨终端的灵活接入，响应REQ-S4-03（B/S架构）。

B/S架构接入方式：
- Web管理端：Vue3/React单页应用，通过API网关访问后端服务，支持Chrome、Firefox、Safari等主流浏览器；
- 微信小程序：调用后端统一的RESTful API，数据通道 HTTPS加密；
- 行业API：外部系统通过API网关接入，采用OAuth2.0鉴权，支撑交通、健康、旅游智能体的标准化接入。

### 2.2.2 技术路线设计

本项目技术路线围绕"物理机理+深度学习融合"这一核心方法展开，分为数据融合处理路线、模型训练路线和平台服务路线三条主线。

**数据融合处理路线**：多源数据经质量控制后进入融合处理管道。气象观测数据（温度、湿度、风速、风向、气压）作为基础输入，数值模式产品（ECMWF、GFS）提供大尺度背景场，卫星遥感数据（风云FYD、MODIS）提供空间覆盖，地表和大气监测数据补充地面实况。各源数据经标准化、质量控制、时间对齐和空间插值后，在特征层面进行融合，生成模型输入特征张量。融合过程采用时空插值和特征交叉方法，保留各源数据的优势信息。

数据融合处理流程的输入输出如下：

| 处理阶段 | 输入 | 处理动作 | 输出 |
|---------|------|---------|------|
| 数据获取 | 各数据源原始数据 | HTTP/FTP拉取或订阅 | 原始数据流 |
| 质量控制 | 原始数据 | 异常值剔除、缺测插值、格式校验 | 质量可控数据 |
| 时间对齐 | 多源时间序列 | 时间戳对齐至统一时间轴 | 对齐后时间序列 |
| 空间插值 | 离散站点数据 | 反距离加权/克里金插值至网格 | 空间网格数据 |
| 特征融合 | 多源网格数据 | 特征交叉、归一化、降维 | 融合特征张量 |

**模型训练路线**：融合特征进入沙尘动态预测AI模型训练管道。本模型采用物理机理引导的深度学习架构，底层网络为时序神经网络（长短期记忆网络LSTM或时序卷积网络TCN），在输入层和损失函数中嵌入沙尘传输的物理约束（如风场驱动、扩散方程）。训练过程使用历史个例数据（宁夏及周边地区历史沙尘个例）进行监督学习，采用迁移学习方法利用少量样本完成模型适配。训练完成后模型以服务化方式部署，接收实时融合特征并输出未来24小时以上沙尘风险空间分布（网格分辨率待确认）和浓度等级。

模型训练流程的步骤、输入、输出如下：

| 训练阶段 | 步骤 | 输入 | 输出 | 责任人/系统 |
|---------|------|------|------|------------|
| 数据准备 | 历史个例筛选、标签标注 | 宁夏历史沙尘个例数据库 | 训练集/验证集/测试集 | 数据工程师 |
| 特征构建 | 融合特征生成 | 多模态原始数据 | 融合特征张量 | 融合处理管道 |
| 模型训练 | 物理约束嵌入、监督学习 | 训练集、验证集 | 模型权重文件 | 模型训练平台 |
| 迁移适配 | 微调适配宁夏数据 | 预训练权重、宁夏个例 | 适配后模型 | 模型工程师 |
| 效果验证 | 准确度/召回率/预警提前量评估 | 测试集 | 评估报告（准确度≥75%） | 模型工程师 |
| 模型部署 | 模型服务化 | 模型权重文件 | REST API推理接口 | 运维工程师 |

**平台服务路线**：模型推理结果推送到平台服务层后，分三路处理——实时预警判断（流式管道）、历史回溯分析（批式管道）和空间制图（GIS管道）。预警判断根据风险阈值生成预警信号并触发发布流程；历史回溯支撑模型效果评估和个例复盘；空间制图生成可视化产品供Web端和小程序展示。三路结果统一写入结果存储层，支撑多渠道调用。

### 2.2.3 技术栈组合

本项目主要技术栈选型如下：

| 技术层级 | 选型 | 版本要求 | 说明 |
|---------|------|---------|------|
| 基础设施 | Docker + Kubernetes (K8s) | 1.25+ | 服务容器化，支撑弹性扩缩容；K8s实现服务注册、健康检查和自动恢复 |
| 关系数据库 | PostgreSQL | 14+ | 存储用户管理、业务配置、预警记录等结构化数据 |
| 时序数据库 | InfluxDB / TimescaleDB | 2.0+ | 存储气象站/监测站时序监测数据，支持时间范围查询 |
| 对象存储 | MinIO (兼容S3协议) | latest | 存储模型文件、训练数据、可视化产品 |
| 流处理引擎 | Apache Flink | 1.17+ | 统一调度流式和批式处理管道 |
| 批处理框架 | Flink Batch + Spark (备选) | 3.0+ | 大规模历史数据离线分析 |
| 深度学习框架 | PyTorch | 2.0+ | 模型训练，支持LSTM/TCN架构 |
| 模型推理加速 | ONNX Runtime | 1.12+ | 模型推理部署，支撑跨框架模型格式转换 |
| GIS服务器 | GeoServer | 2.22+ | 空间数据发布，WMTS/WMS服务 |
| 空间数据库 | PostGIS | 3.3+ | 空间数据存储和查询 |
| API网关 | Kong / Nginx Ingress | 3.0+ | 统一鉴权、限流、日志、路由 |
| 前端框架 | Vue3 + Element Plus | 3.0+ | Web管理端 |
| 小程序框架 | 微信小程序原生 | latest | 移动端用户入口 |
| 监控告警 | Prometheus + Grafana | - | 服务指标采集和可视化 |
| 日志分析 | ELK (Elasticsearch/Logstash/Kibana) | 8.0+ | 日志统一采集和检索 |

技术栈选型遵循以下原则：
- 主流开源优先，避免绑定单一厂商；
- 社区活跃度高，文档完善，长期维护有保障；

### 2.2.5 关键技术难点与应对

本项目技术路线面临以下关键难点：

**物理机理与深度学习融合**：单一深度学习模型在沙尘预报中易出现过拟合和泛化能力不足问题。本项目采用物理机理引导的损失函数设计，在常规交叉熵或均方误差损失基础上增加风场驱动约束和连续性约束，使模型在物理上更合理。应对措施为在模型训练前开展宁夏历史沙尘个例的物理规律分析，并据此设计约束项。

**小样本模型训练**：宁夏地区沙尘天气个例有限，纯数据驱动方法面临样本不足问题。本项目采用迁移学习方法，利用已有全国或更大范围的沙尘模型权重进行微调（fine-tuning），结合数据增强（时间序列增强、空间扰动）扩大训练样本集。

**流批一体调度复杂性**：实时预警和离线分析共用同一套处理引擎，增加了调度复杂性。本项目采用Flink作为统一引擎，通过作业链（Job Graph）管理流式和批式任务的依赖关系，设置独立的作业监控和异常告警。

### 2.2.6 与前后章节的技术接口

本节技术架构与技术路线在以下节点与前后章节形成接口约束：

- **与2.1总体架构设计的接口**：本节在微服务、流批一体、B/S三种架构模式基础上细化各服务的选型和组合；各服务职责划分须与2.1.5模块职责边界表格保持一致。
- **与2.3接口设计的接口**：本节定义的API网关和三条技术路线（数据融合/模型训练/平台服务）构成2.3接口设计的约束基础；2.3须在REST API清单中覆盖本节各服务的接口。
- **与2.4部署架构的接口**：本节技术栈选型（Docker/K8s、PostgreSQL、PostGIS、Flink等）构成2.4部署架构的资源配置约束；2.4须在服务器/容器配置中覆盖本节技术栈。
- **与3.1.2模型训练的接口**：本节模型训练路线（物理机理+深度学习融合、迁移学习）直接支撑3.1.2节的模型构建内容；3.1.2须在本节技术路线约束下细化模型架构。
