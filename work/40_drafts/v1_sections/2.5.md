### 2.5.2 性能设计

#### 2.5.2.1 性能指标体系与责任边界

性能指标体系以用户可感知的端到端响应时间和系统并发吞吐量为两条主线，各指标分解至对应系统层级，明确端到端性能的责任边界。

**核心性能指标分解矩阵：**

| 指标项 | 量化要求 | 测量方式 | 责任人 |
|-------|---------|---------|-------|
| 平台全链路响应（第95百分位） | ≤3秒 | JMeter模拟真实用户操作路径 | 平台服务层负责人 |
| 前端页面首次交互（FID） | ≤2秒 | Lighthouse / Chrome DevTools Performance | 前端负责人 |
| 沙尘预测结果单次查询响应 | ≤2秒 | API端到端测试（含缓存路径） | 后端API负责人 |
| 预警信息从模型输出到终端展示 | ≤5秒 | 端到端时延监控（模型→消息队列→推送→前端） | 系统集成负责人 |
| 并发接入上限 | 100用户/50并发 | JMeter阶梯加压测试 | 平台架构负责人 |
| 系统吞吐量上限 | 待结合实际数据量确认（MANUAL） | LoadRunner持续加压至性能拐点 | 平台架构负责人 |

**性能设计原则：**

1. **预核算**：设计阶段以REQ-S2-03/04/05为硬性约束，按50并发的1.5倍（即75并发）进行容量规划，确保峰值下仍有充足余量。
2. **分层负责**：每项端到端指标分解至各层级组件，任何一层未达标均触发责任追溯和优化。
3. **可降级**：在系统承压时，按"核心功能优先于辅助功能"原则，先降级非关键组件（如GIS空间分析高亮动画），再限制非核心接口。
4. **可验证**：每项性能指标对应明确测试方法和通过标准，不接受"优化后应该能达到"等模糊承诺。

#### 2.5.2.2 前端性能保障方案

前端性能设计针对"前端交互2秒内"目标，从网络层、渲染层、缓存层三个层面进行保障：

**网络层**

- **资源压缩与合并**：前端JS/CSS资源经Webpack构建后启用Gzip压缩（压缩率≥70%），同类型多个请求合并为一条，减少HTTP请求数量。
- **CDN加速静态资源**：图片、字体、第三方库等静态资源托管至CDN就近节点，首屏资源加载时间控制在500ms以内。
- **HTTP/2多路复用**：生产环境启用HTTP/2协议，替代HTTP/1.1的管道化请求模式，减少往返时延（RTT）。

**渲染层**

- **首屏优先渲染策略**：HTML文档流解析时优先加载首屏内容（核心地图图层 + 预测结果显示卡片），非首屏内容延迟加载，首屏内容在1.5秒内完成展示。
- **骨架屏（Skeleton Screen）**：数据加载期间展示内容骨架占位，避免用户感知"白屏"。
- **虚拟列表**：当预测结果列表超过100条时，前端使用虚拟列表技术，只渲染当前视口可见行，降低DOM节点数和重绘成本。
- **懒加载组件**：非首屏功能模块（场景分析面板、GIS工具栏等）按需动态加载，主bundle体积控制在≤500KB（压缩后）。

**缓存层**

- **Service Worker缓存**：前端静态资源（JS/CSS/图片）通过Service Worker缓存，支持离线访问和资源预取。
- **IndexedDB历史数据缓存**：最近一次沙尘预测全量结果存储于IndexedDB，重复访问直接读取本地缓存，API调用量减少≥60%。
- **内存缓存**：前端内存中缓存当前会话数据，避免重复计算和请求。

**前端性能验收标准：**

| 验收指标 | 工具/方法 | 通过标准 |
|---------|---------|---------|
| First Input Delay（FID） | Lighthouse | ≤2秒（第95百分位） |
| Largest Contentful Paint（LCP） | Lighthouse | ≤2.5秒 |
| 静态资源加载时间 | Chrome DevTools Network | 主资源（HTML + 关键CSS/JS）≤800ms（首次访问，CDN缓存命中后） |
| IndexedDB缓存命中率 | 前端埋点日志统计 | 重复访问时缓存命中≥60% |

#### 2.5.2.3 后端性能保障方案

后端性能设计以REQ-S2-03（界面响应3秒内）和REQ-S2-05（100用户/50并发）为硬性目标，从接入层、业务层、数据层逐层保障：

**接入层（API网关）**

- **智能限流**：API网关（Kong/AWS API Gateway或等效方案）设置全局限流策略——每秒请求数（QPS）上限100，单IP最大并发连接数50，超出部分进入FIFO排队队列（最大排队长度200，超过则返回429 Too Many Requests）。此策略同时满足REQ-S2-05的容量约束。
- **连接复用与HTTP/2**：网关与后端服务间启用HTTP/2长连接，复用TCP连接，减少建连开销。
- **请求校验前移**：不合规请求（如参数格式错误、无效Token）在网关层直接拦截并返回，减少后端处理资源消耗。

**业务逻辑层**

- **读写分离架构**：数据库主库承担写入和实时查询，从库承担只读查询（如历史预测结果翻页、历史统计图表）。主从复制延迟控制在≤100ms，从库查询无需关注一致性视图问题。
- **异步处理队列**：复杂业务逻辑（如沙尘影响评估计算、GIS空间分析）封装为异步任务，提交后立即返回任务ID，客户端通过轮询或WebSocket接收结果。异步任务队列使用Redis实现，队列满时触发告警并启动扩容流程。
- **热点数据缓存**：预测结果（按站点编码+时间戳为键）、用户配置信息、权限信息等热点数据缓存至Redis，TTL设置为30分钟（预测结果）和60分钟（配置信息），缓存命中率目标≥70%。
- **模型推理加速**：模型推理服务支持GPU加速（NVIDIA T4或等效GPU），单次推理时延目标≤500ms（含模型加载和数据预处理）。

**数据存储层**

- **连接池管理**：数据库连接池大小按50并发的1.5倍规划（75个连接），单次查询超时设置为500ms，慢查询（>1秒）自动记录并告警。
- **索引优化**：针对高频查询字段（站点编码、预测时间、用户ID）建立复合索引，避免全表扫描。查询计划每月审查一次。
- **冷热数据分离**：最近30天热数据存储于SSD云盘（高IOPS），30天以前的历史数据自动归档至对象存储（成本优先），归档数据查询响应时间目标≤10秒（MANUAL——如用户有快速查询历史数据的需求，需调整归档策略）。

**后端性能验收标准：**

| 验收指标 | 测试方法 | 通过标准 |
|---------|---------|---------|
| API第95百分位响应时间 | JMeter模拟50并发持续30分钟 | ≤3秒 |
| 50并发时请求成功率 | JMeter阶梯加压（10→30→50） | ≥99% |
| Redis缓存命中率 | Redis INFO stats统计 | ≥70% |
| 异步任务处理时效 | 提交10个异步任务，记录从提交到完成的时间 | 90%任务在≤60秒内完成 |
| 数据库连接池健康度 | 监控连接池活跃/等待/空闲连接数 | 无连接等待超时，无连接泄漏 |

#### 2.5.2.4 全链路性能监控与闭环

性能保障不止于设计阶段，还需通过上线后的持续监控形成闭环：

1. **APM全链路追踪**：在各微服务部署APM Agent（SkyWalking/Elastic APM或等效方案），每个请求携带trace_id，串联从网关到数据库的全链路耗时数据。
2. **性能基线与告警**：上线稳定后建立性能基线（各类API平均响应时间、缓存命中率等），设置动态阈值告警——当响应时间超过基线150%且持续5分钟，触发值班告警。
3. **定期性能评审**：每月输出性能运行报告，包含响应时间趋势、缓存命中率变化、慢查询TOP10，跟踪优化。

#### 2.5.3.2 网络与边界安全

| 安全措施 | 具体做法 | 防护目标 |
|---------|---------|---------|
| 南北向流量控制 | API网关强制HTTPS（TLS 1.2+），域名白名单校验，请求体大小限制（单请求≤10MB），异常IP自动封禁 | 防止外部未授权访问和DDoS |
| 东西向服务通信加密 | 微服务间通信强制TLS 1.2+，服务间调用携带短期有效JWT Token（有效期≤2小时） | 防止内部横向渗透 |
| DMZ隔离 | 前端Web服务部署于DMZ区，模型推理服务、数据库部署于内网区，两区通过API网关通信 | 保护核心数据资产，减小攻击面 |
| 网络ACL策略 | 按最小权限配置安全组/防火墙ACL，默认拒绝，仅开放业务必要端口（如443/80） | 减少暴露端口，阻止非法扫描 |
| WAF基础防护 | API网关前部署WAF，防护SQL注入、XSS、路径遍历等常见Web攻击 | OWASP Top 10防护 |

#### 2.5.3.3 应用安全

| 安全措施 | 具体做法 | 对应采购要求 |
|---------|---------|------------|
| 身份认证 | 统一身份认证平台（或等效方案），JWT Token有效期2小时，RefreshToken有效期14天，错误密码连续5次触发账户锁定30分钟 | REQ-S4-05 |
| 授权管理（RBAC） | 基于角色的访问控制，普通用户仅可访问本人有权限的数据，管理员可配置用户角色和权限，支持细粒度到接口级别的权限控制 | REQ-S4-05 |
| 敏感数据脱敏 | 手机号、身份证号等PII字段在前端展示和日志记录时自动脱敏（仅显示首尾各3位） | REQ-S4-05个人信息保护 |
| SQL注入防护 | 所有数据库查询使用参数化查询，禁止字符串拼接构建SQL语句 | OWASP Top 10 |
| XSS防护 | 前端输入内容进行HTML转义，后端设置Content-Security-Policy响应头 | OWASP Top 10 |
| 接口防重放攻击 | 请求携带时间戳（允许误差±5分钟）和随机nonce，5分钟内相同nonce的重复请求拒绝 | 防止重放攻击 |
| 敏感操作审计 | 用户登录、数据导出、数据删除等敏感操作记录审计日志（含操作人、操作时间、操作内容、操作结果） | REQ-S4-05 |
| 会话安全管理 | JWT Token失效机制、同一账号最大并发会话数限制（5个）、会话超时自动登出（30分钟无操作） | 防止会话劫持 |

#### 2.5.3.4 数据安全

| 数据类别 | 安全措施 | 存储位置与访问控制 |
|---------|---------|-----------------|
| 沙尘预测原始数据 | 传输过程TLS加密，存储时AES-256加密，每日增量备份+每周全量备份，备份数据单独加密 | 宁夏气象局内网服务器，仅模型服务有解密读取权限 |
| 用户业务数据 | 传输加密，存储加密，访问需日志记录，用户数据导出需二次确认和审批 | 内网数据库，管理员访问需记录审计日志 |
| 系统配置信息 | 配置值加密存储，不同环境配置隔离（开发/测试/生产），配置变更需操作记录 | 配置中心，配置管理员通过审批流程修改 |
| 预测模型文件 | 模型文件存储于内网隔离仓库（harbor/nexus），下载需身份认证，模型版本与SHA256指纹绑定 | 内网模型仓库，模型管理员维护 |
| 备份数据 | 备份文件单独加密存储，加密密钥与数据密钥分离，密钥通过KMS管理 | 异地存储卷，仅备份恢复场景可解密 |

#### 2.5.3.5 安全验收方式

| 验收项目 | 验收方式 | 判定标准 |
|---------|---------|---------|
| 身份认证功能 | 使用正确账号密码登录成功；错误密码连续5次触发账户锁定；锁定期间正确密码无法登录 | 登录成功，锁定机制生效 |

| XSS防护 | 在输入框提交`<script>alert(1)</script>`，验证前端未执行该脚本 | XSS攻击被转义或过滤 |

| 审计日志可追溯 | 通过trace_id追溯用户登录→数据查询→数据导出的完整操作链路 | 审计日志链路完整 |
| 渗透测试（MANUAL） | 委托第三方安全机构进行黑盒渗透测试（待用户确认预算后执行） | 无高危漏洞，修复后由用户确认 |

#### 2.5.4.2 故障分类与响应矩阵

| 故障等级 | 定义 | 响应时效 | 通知范围 |
|---------|------|---------|---------|
| P0（重大故障） | 平台完全不可用，或核心功能（预测展示、预警推送）不可用超过15分钟 | 5分钟告警，15分钟响应，1小时定位 | 项目经理 + 技术负责人 + 用户方接口人 |
| P1（严重故障） | 核心功能可用但性能显著下降（如响应时间>10秒），或非核心功能不可用 | 15分钟告警，30分钟响应，4小时恢复 | 技术负责人 + 值班人员 |
| P2（一般故障） | 单个用户/单点问题，不影响全局服务 | 下一工作日处理 | 值班人员记录跟踪 |

#### 2.5.4.3 服务级容错设计

**微服务健康检查与自动摘除**

各微服务每15秒向服务注册中心（Nacos/Consul或等效方案）发送心跳，连续3次无心跳则标记为不健康。API网关将该节点从可用节点池中移除，正在处理的请求在当前请求处理完毕后终止，不再分发新请求。新请求自动路由至健康节点，全过程对用户无感知。

**服务多副本部署**

- API网关：至少2个节点，分布于不同可用区（AZ），单AZ故障不影响服务。
- 平台核心服务（预测查询服务、用户服务）：至少2个副本，无状态设计，通过负载均衡分发请求。
- 模型推理服务：1主+1备GPU副本，主副本故障时自动切换至备用副本，切换时间≤30秒（MANUAL——GPU切换方案待技术验证后补充）。

**熔断降级机制**

当模型推理服务出现以下任一情况时，API网关触发熔断，后续推理请求返回降级响应：

- 连续3次推理超时（单次超时阈值=5秒）
- 错误率超过10%（过去5分钟内5xx比例）
- 推理服务不可用（健康检查连续失败）

熔断触发后，每隔30秒放行1个探测请求试探服务是否恢复，服务恢复后熔断自动解除。降级期间前端展示"预测服务暂时不可用，请稍后刷新"的友好提示，不影响其他功能使用。

**异步队列保护**

异步任务队列（Redis Stream或Kafka）设置最大队列长度（10000条），队列超过80%容量时触发告警，队列满时新任务拒绝并返回前端错误码，防止队列积压导致的服务崩溃。

#### 2.5.4.4 数据级容错设计

**冷热数据分离**

| 数据类型 | 存储介质 | 保留策略 | 查询响应目标 |
|---------|---------|---------|------------|
| 热数据（最近30天预测结果） | SSD云盘（高IOPS） | 实时写入，30天后自动归档 | ≤2秒 |
| 温数据（31~180天历史数据） | 普通云盘 | 归档后压缩存储，支持查询 | ≤10秒（MANUAL） |
| 冷数据（180天以前） | 对象存储（成本优先） | 归档存储，仅保留元数据索引 | 需要时恢复后提供 |

> **MANUAL——历史数据查询需求**：如用户有快速查询180天以前历史数据的业务需求，需调整冷热分离策略和索引方案，待用户确认后补充。

**备份与恢复策略**

| 备份类型 | 频率 | 保留时长 | 恢复时效目标 |
|---------|------|---------|------------|
| 实时写入（WAL归档） | 实时 | 与主数据一致 | RPO≈0 |
| 增量备份 | 每小时 | 72小时 | RTO≤1小时 |
| 全量备份 | 每日凌晨2点 | 30天 | RTO≤4小时 |
| 异地容灾备份（MANUAL） | 每日 | 30天 | RTO≤24小时（待确认容灾方案） |

**跨AZ容灾（MANUAL）**

如用户确认需要跨地域容灾能力（宁夏气象局有此要求），可将备用副本部署至银川市另一个可用区，RTO目标≤24小时，预算和详细方案待确认后补充。

#### 2.5.4.5 可靠性验收方式

| 验收项目 | 验收方式 | 判定标准 |
|---------|---------|---------|
| 服务多副本可用性 | 手动关闭任一服务副本，观察请求是否自动路由至健康副本 | 服务不中断，用户无感知 |
| 熔断触发有效性 | 模拟模型推理服务故障（返回500），验证熔断触发；故障恢复后验证自动解除 | 熔断在1分钟内触发，30秒内自动恢复 |
| 数据备份可恢复性 | 从最新备份恢复至测试环境，验证数据完整性和业务功能可用 | 恢复后数据完整，功能正常 |

### 2.5.5 可运维性与监控设计

#### 2.5.5.1 运维监控体系架构

本项目运维监控体系覆盖基础设施监控、应用性能监控（APM）和业务指标监控三个层面，参考Google SRE的"Golden Signals"方法论，以量化指标驱动运维决策。

**监控数据采集与存储架构**

```
[采集层]
  指标采集 → Prometheus（node_exporter / cAdvisor / APM agents）
  日志采集 → Filebeat → Logstash → Elasticsearch
  链路追踪 → Jaeger agent → Jaeger Collector
       ↓
[存储层]
  时序数据 → Prometheus TSDB（或M3DB）
  日志数据 → Elasticsearch（180天留存）
  链路数据 → Jaeger（7天留存）
       ↓
[展示层]
  指标大盘 → Grafana（可自定义仪表板）
  日志检索 → Kibana
  调用链路 → Jaeger UI
       ↓
[告警层]
  告警聚合 → Prometheus AlertManager
  告警通知 → 微信（主）/ 短信（备）/ 邮件
  告警升级 → 按故障等级自动升级通知范围
```

#### 2.5.5.2 核心监控指标（Golden Signals）

| 指标类别 | 具体指标 | 告警阈值（示例） | 采集频率 |
|---------|---------|---------------|---------|
| 延迟（Latency） | API第95百分位响应时间 | >3秒持续5分钟 | 15秒 |
| 流量（Traffic） | QPS / 并发连接数 | 接近容量80%持续10分钟 | 15秒 |
| 错误（Errors） | HTTP 5xx错误率 | >1%持续5分钟 | 15秒 |
| 饱和度（Saturation） | CPU使用率 / 内存使用率 / GPU利用率 | >85%持续10分钟 | 15秒 |
| 业务指标 | 预测准确度低于阈值、数据更新延迟超30分钟 | 准确度<70%或数据延迟>30分钟 | 1小时（模型质量评估） |
| 模型推理时延 | 单次推理P99时延 | >2秒持续5分钟 | 每次推理记录 |
| 队列积压 | 异步任务队列长度 | 队列长度>80%容量持续5分钟 | 1分钟 |

#### 2.5.5.3 日志管理规范

| 日志类型 | 内容 | 格式 | 保留策略 | 用途 |
|---------|------|------|---------|-----|
| 访问日志 | 所有API请求的时间、来源IP、请求参数、响应码、响应时间 | JSON结构化 | 180天 | 安全审计、问题排查、访问量分析 |
| 业务日志 | 业务操作记录，含trace_id、用户ID、操作类型、操作结果 | JSON结构化 | 90天 | 业务问题排查、操作追溯 |
| 系统日志 | OS层和容器运行日志（/var/log、journald） | 非结构化文本 | 90天 | 系统故障根因分析 |
| 审计日志 | 登录、数据查询、数据导出、数据删除等敏感操作 | JSON结构化，含操作人/时间/内容/结果 | 360天（满足合规要求） | 安全合规审计 |
| 模型推理日志 | 推理请求时间戳、模型版本、输入参数、推理耗时、输出摘要 | JSON结构化 | 180天 | 模型性能优化和质量追溯 |

> **MANUAL——审计日志留存期**：以上审计日志360天留存期为行业通用合规要求参考值。如宁夏气象局有特定合规要求（如等保2.0要求日志留存≥6个月），以用户书面确认为准。

#### 2.5.5.4 运维操作规程

**版本发布规程**

1. **环境准备**：在发布前，CI/CD流水线自动执行单元测试、集成测试、代码扫描，测试通过方可进入发布审批。
2. **灰度发布**：先在测试环境全量验证，再将新版本部署至10%生产节点，观察24小时（至少覆盖一个完整的预测周期）无异常后，再全量升级至剩余90%节点。
3. **回滚机制**：每次发布保留上一稳定版本镜像和配置。发现异常后，运维人员在CI/CD平台上执行回滚操作，15分钟内完成回滚至上一稳定版本。全过程不超过10分钟。
4. **发布窗口**：非紧急版本发布安排在凌晨00:00至06:00进行，避免影响用户日间使用。

**配置变更规程**

1. 配置变更统一通过配置中心（Apollo/Nacos或等效方案）管理，不允许直接修改配置文件。
2. 配置变更前需在测试环境验证，变更审批流程包含：申请人→技术负责人→用户确认（影响核心参数时）。
3. 配置变更记录自动生成审计日志，支持追溯变更人、变更时间、变更前后的配置值。
4. 重要配置变更（如限流阈值、熔断参数）变更后需持续观察24小时，确认无异常。

**故障响应规程**

| 步骤 | 操作 | 时限 |
|------|------|------|
| 1. 故障发现与告警 | 监控系统检测到异常，自动发送告警至值班人员 | 5分钟内 |
| 2. 值班响应 | 值班人员确认故障，评估等级（见2.5.4.2故障分级） | P0: 15分钟内；P1: 30分钟内 |
| 3. 问题定位 | 通过Grafana大盘 + Kibana日志 + Jaeger链路逐步定位根因 | 1小时内给出初步定位 |
| 4. 故障处置 | 按故障类型执行熔断/重启/回滚等应急操作 | 不超过30分钟（单一操作） |
| 5. 恢复验证 | 确认服务恢复，通过监控验证指标回归正常 | 恢复后15分钟内 |
| 6. 事后复盘 | 编写故障报告（含根因、处置过程、改进措施），提交用户方 | 3个工作日内 |

**日常巡检规程**

- 每周巡检：系统资源使用情况（CPU/内存/磁盘/GPU）、告警统计、备份执行情况、登录异常检查。输出周巡检报告。
- 每月巡检：在周巡检基础上增加安全扫描（漏洞扫描+配置审计）、数据库性能审查、慢查询优化、容量预测（基于趋势分析预测未来3个月资源需求）。输出月巡检报告。

#### 2.5.5.5 可运维性验收方式

| 验收项目 | 验收方式 | 判定标准 |
|---------|---------|---------|
| 监控大盘可访问性 | 通过Grafana URL访问主监控大盘，确认所有Golden Signals指标正常展示 | 所有指标可见且数据实时更新 |
| 告警可触达性 | 模拟指标超阈值场景，验证告警通过预设渠道（微信/短信）发送 | 告警在5分钟内送达指定人员 |
| 日志可追溯性 | 选取一次业务请求，通过trace_id在ELK中追溯完整调用链路 | 链路完整，可定位问题 |

| 日常巡检可执行性 | 执行一次完整的周巡检流程，输出巡检报告 | 巡检报告完整，包含所有检查项 |
