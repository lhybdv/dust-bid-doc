# 2 总体技术方案

## 2.1 总体架构设计

### 2.1.1 架构设计目标与原则

本项目以沙尘动态预测AI模型训练为基础，以沙尘监测预警示范服务场景应用平台为载体，构建覆盖公众、交通、人体健康、旅游等场景的沙尘影响智能评估与服务转化能力。方案完整响应采购文件关于预测提前量、准确度和平台性能的硬性要求，并突出预测结果向风险等级、行业提示、公众建议和联动接口转化的应用价值。

本项目采用微服务架构、流批一体化框架和B/S架构模式，整体系统划分为模型训练层、平台服务层和场景服务转化层三层结构，各层职责明确，层间通过标准化接口通信。

### 2.1.2 整体架构

系统整体采用微服务架构，按业务边界拆分为独立部署的服务单元，各服务可通过统一API网关对外暴露。平台服务端采用B/S架构，浏览器侧为Web应用，移动侧为微信小程序，外部系统通过RESTful API接入。

- **场景服务转化层**将平台能力输出至Web管理端、微信小程序、行业API和三领域智能体，完成面向交通、人体健康、旅游和公众的服务闭环，响应REQ-S3-03、REQ-S3-04多场景应用要求。

### 2.1.3 技术架构特点

本方案技术架构满足以下要求：

**微服务架构**：系统按功能边界拆分为独立服务单元，包括预测服务、实况监测服务、预警服务、场景服务、GIS服务、用户管理服务等，各服务独立部署、独立扩展，通过服务注册与发现机制协调。

**流批一体化框架**：实时数据（气象站、监测站、能见度）采用流式处理管道，模型推理和历史回溯采用批处理管道，两套管道共享特征存储和模型存储层，统一由流批一体处理引擎调度。

**B/S架构**：用户侧通过浏览器访问Web管理端，移动用户通过微信小程序访问，后台采用JSON over HTTPS标准接口，支持跨平台调用。

**扩展性设计**：平台各模块采用松耦合设计，新增场景或行业接口仅需在API网关注册，无需改动核心服务。

### 2.1.4 模块职责边界

| 模块 | 职责 | 输入 | 输出 |
|------|------|------|------|
| 预测服务 | 接收数据融合结果，执行模型推理 | 融合特征数据流 | 风险等级空间分布 |
| 实况监测服务 | 采集并展示实时监测数据 | 气象站/监测站/遥感数据 | 实时监测页面数据 |
| 预警服务 | 制作、审核和发布预警产品 | 预测结果+阈值配置 | 预警产品+发布记录 |
| 场景服务 | 生成交通/健康/旅游场景建议 | 风险等级+场景参数 | 场景服务建议 |
| GIS服务 | 提供空间分析和图关系能力 | 空间数据+图关系请求 | 空间分析结果 |
| API网关 | 统一入口，鉴权、限流、日志 | 各渠道请求 | 标准化API响应 |

### 2.1.5 验收依据

本节架构设计通过以下交付物验收：

1. **架构设计说明书**：包含三层架构图、服务划分表、接口列表，提交时间为合同签订后30个工作日内。
2. **平台原型演示**：模型训练层、平台服务层和场景服务转化层的联调演示，响应REQ-S4-01（微服务）、REQ-S4-02（流批一体）、REQ-S4-03（B/S）架构要求。
3. **部署验收测试**：在宁夏回族自治区气象局机房完成部署后，开展功能验收和性能验收（界面响应≤3秒，前端交互≤2秒，50并发稳定运行），验证REQ-S2-03、S2-04、S2-05、S2-06指标达成情况。
