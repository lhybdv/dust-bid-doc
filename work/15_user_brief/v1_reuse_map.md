# v1 旧稿复用地图

## 一、总体判断

`work/40_drafts/v1.md` 不适合作为新标书的正文底稿继续修改。主要原因是它已经被写成一份完整长稿，结构和叙事惯性较强，继续修会很容易沿着旧方向越写越重。

但旧稿并不应废弃。它包含大量可复用素材，尤其是商务条款、硬性指标、平台模块、实施进度、测试、运维、培训和售后内容。新一轮重写应把它当作素材库：抽取有用内容，重新组织，不继承旧稿的问题。

| 旧稿位置 | 可复用内容 | 使用方式 |
|----------|------------|----------|

| A5 商务条款偏离表 | 24个月、服务地点、团队5人以上、培训、知识产权、付款、售后服务 | 可直接作为商务响应表基础 |

## 五、应避免继续使用的内容

| 类型 | 说明 |
|------|------|
| “主标结果接入”式依赖表达 | 当前口径为副标独立完整方案，不能写成依赖竞争方或主标输出 |
| “智能体自研”式表达 | 已确认三领域智能体由宁夏移动建设，本项目通过接口接入 |
| 过度宏大的核心预测模型叙事 | 采购要求要响应，但新稿优势应落在预测结果如何被业务使用 |
| 空表格和大量待填写 | 正式商务部分可以有占位，但技术正文不要靠占位撑结构 |
| 重复口号 | 如“全面提升、智能化赋能、构建完整体系”等，应改成具体动作和指标 |
| 过长索引 | 索引应服务评审，不要先于正文变成篇幅负担 |

## 六、按章节重写时的取材建议

### 商务部分

从 A2、A5、A6、B1 取硬性承诺和表格框架。不要保留大段模板式企业实力说明，除非用户补充真实资质、业绩和人员信息。

### 项目实施方案

从 B2 取 24个月五阶段计划：

1. 第1-3个月：需求分析；
2. 第4-6个月：系统设计；
3. 第7-16个月：模型训练、平台开发、小程序和API开发；
4. 第17-21个月：部署测试、初验和试运行；
5. 第22-24个月：整改、终验、培训和正式交付。

新稿中应补充每阶段交付物和验收依据，减少横向展开。

### 模型技术方案

从 B3 只取素材，不取旧结构。建议新结构：

1. 建设目标与指标响应；
2. 数据集与多模态数据处理；
3. 沙尘动态预测AI模型训练；
4. 沙尘影响智能评估与服务转化模型；
5. 交通、健康、旅游、公众场景风险分级；
6. 自研图关系辅助分析；
7. 宁夏移动智能体接口接入；
8. 模型检验评估和应用验证。

### 平台研发方案

从 B4 取功能模块。重点保留：

- 实况监测；
- 预报预警；
- 场景服务分析；
- 智能体服务集成；
- GIS；
- 微信小程序；
- 行业API；
- 系统管理和运维监控。

新稿应把“平台如何承接模型输出并服务用户”写清楚。

### 性能、运维、培训、售后

从 B5、B6、B7 复用，并统一为可验收指标：

- 界面响应 <=3秒；
- 前端交互 <=2秒；
- 100用户、50并发；
- 7*24不间断服务；
- 30分钟实质性响应；
- 1小时远程诊断；
- 24小时现场支持；
- 不少于2次培训；
- 2年免费运维。

## 七、旧稿最有价值的素材清单

- 平台采用微服务架构、流批一体化框架，支持Web端、微信小程序端、行业API接口；
- 实况监测融合气象站、环境监测站、卫星遥感、能见度、PM10/AQI等数据；
- 场景服务覆盖交通、人体健康、旅游和公众服务；
- 交通场景输出道路通行风险、限速建议和重点路段提示；
- 健康场景输出敏感人群防护等级、健康风险提示和防护建议；
- 旅游场景输出景区影响等级、出行适宜性和行程调整建议；
- 小程序提供实时能见度地图、风险提醒、个性化防护建议和订阅推送；
- 行业API提供风险查询、预警订阅、场景结果和智能体分析结果接口；
- 运维方案包含监控、告警、备份、安全防护和故障响应流程；
- 培训方案覆盖模型应用、平台操作、运维管理三类人员。

## 八、下一轮写作提示

重写时不要问“旧稿哪段能直接搬”。更好的问法是：

1. 这一章要响应哪个采购要求？
2. 这一章要拿哪个评分点？
3. 旧稿里有哪些素材能支撑这个要求？
4. 哪些旧表述会把方案带回错误方向？

按这个顺序使用 `v1.md`，旧稿就不会浪费，也不会继续拖着新稿往旧问题里走。
